# Failure, Stop, and Return

このドキュメントは、  
BOA Core が **「失敗」や「停止」をどう扱うか** を説明します。

ここで言う失敗とは、
例外処理やエラーハンドリングの話ではありません。

---

## この設計における「失敗」とは何か

BOA Core において、  
最も避けるべき失敗は次の状態です。

> **判断できないのに、判断が進んでしまうこと**

- 情報が足りない
- 文脈が崩れている
- 前提が変わっている
- 責任の所在が曖昧

これらの状態で判断が行われること自体が、  
この設計における「失敗」です。

---

## Stop は異常ではない

BOA Core では、  
**Stop（停止）は正常系**です。

### BOA における Stop

- 境界条件を満たしていない
- 判断を許可していない領域に入った

この場合、  
判断は **開始されません**。

これは拒否でも失敗でもなく、  
**設計通りの終了**です。

---

## Return は責任ある動作である

RCA や RP が返す Return は、  
判断の放棄ではありません。

Return が意味するのは：

- 今は決められない
- この条件では決めるべきではない
- 人や設計に戻すべきである

という **明示的な意思表示**です。

Return は、
「判断不能を隠さない」ための正規ルートです。

---

## なぜ「黙る」ことを許さないのか

BOA Core では、  
以下の状態は明確に禁止されています。

- 何も返さない
- 暗黙に成功とみなす
- 失敗をログにだけ残して進む

沈黙は、
判断が行われたのかどうかを不明確にします。

不明確な判断は、
**責任を消します。**

### 判断不能は「能力不足」ではない

RCA が UNKNOWN を返す理由は、  
AIの能力不足とは限りません。

- 入力が揃っていない
- 文脈が時間的に破綻している
- 前提条件が設計時と異なる
- 責任境界が不明確

これらは、  
どれだけ賢いモデルでも解決できない問題です。

BOA Core は、  
それを「頑張って解決しない」設計を選びます。

## なぜ止まる設計が必要なのか

AIを含むシステムは、  
動き続けることが評価されがちです。

しかし、判断を扱うシステムでは、

- 早く動くこと
- 多く決めること
- 自動化すること

よりも、

> **正しく止まれること**

の方が重要です。

BOA Core は、  
この価値観を構造として固定します。

## 責任と停止の関係

判断が止まったとき、  
RCA は次を残します。

- なぜ止まったのか
- どこで止まったのか
- 誰が止めたのか（人／AI／設計）

これにより、  
停止は「失敗」ではなく、  
次の判断の材料になります。

## まとめ

- Stop は異常ではない
- Return は責任ある判断である
- 判断不能は隠さない
- 沈黙は許されない
- **止まれることが、安全である**

BOA Core は、  
判断を進めるための設計ではなく、  
判断を壊さないための設計です。
